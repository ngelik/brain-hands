# Hands Self-Review and Ordered Reviewer Actions Design

## Summary

Brain Hands will add a configurable pre-Verifier quality gate and a durable,
ordered fix queue for Reviewer actions. After every successful Hands mutation,
the runtime runs deterministic verification followed by fresh Hands
self-review/fix passes. New configurations use two passes. Only after those
passes finish does control move to an independent Verifier.

When the Verifier requests changes, each finding becomes an immutable ordered
action. Hands fixes one action at a time and cannot see or start later actions
until deterministic verification, the configured self-review passes, and a
focused Verifier resolution check prove the current action resolved.

## Goals

- Make Hands inspect and correct its own work twice before independent review.
- Use fresh ephemeral self-review invocations to reduce confirmation bias.
- Ensure Reviewer actions are processed strictly one by one.
- Define “resolved” through independent evidence rather than a Hands claim.
- Preserve the existing three-cycle fix budget and one backup recovery tier.
- Apply usage-limit fallback to mutation and self-review invocations.
- Persist every queue, action, pass, verification, and resolution boundary so
  resume is idempotent.
- Make the maximum incremental model-call cost visible before queue execution.

## Non-goals

- Allowing Hands to approve its own delivery.
- Dynamically extending any retry or self-review limit.
- Running multiple Reviewer actions concurrently.
- Letting self-review widen the approved work item or active Reviewer action.
- Replacing the full Verifier review after a completed action queue.
- Treating operational failures as implementation findings.

## Considered Approaches

### Repeat one prompt inside the original Hands invocation

This is cheap but does not create an independent inspection context. The same
invocation is likely to repeat its assumptions and cannot provide a durable
pass boundary.

### Fresh self-reviews with all Reviewer findings fixed as one batch

This improves inspection but allows findings to interfere, makes resume
ambiguous, and cannot enforce “do not advance until the previous action is
solved.”

### Fresh self-reviews plus an ordered action queue

This is selected. Each mutation receives fresh self-review passes. Reviewer
findings become immutable actions, and a focused Verifier must resolve the
active action before the next action unlocks.

## Configuration

```yaml
retry_policy:
  max_hands_fix_attempts: 3
  max_replan_attempts: 2
  quality_gate:
    hands_self_review_passes: 2
    max_attempts_per_reviewer_action: 2
    require_focused_verifier_confirmation: true
```

Rules:

- `hands_self_review_passes` accepts integers from `0` through `3`.
- New configuration generated by `brain-hands init` uses `2` passes.
- Existing v2 configuration without `quality_gate` retains the previous
  behavior, equivalent to zero self-review passes and no ordered-action gate.
- `max_attempts_per_reviewer_action` accepts integers from `1` through `3`.
- `require_focused_verifier_confirmation` must be `true` when the ordered-action
  gate is enabled. `false` is rejected rather than silently weakening the
  resolution definition.
- Limits are snapshotted into the run intake and never changed by resume.
- Self-review uses the currently active Hands route. If availability fallback
  has activated backup, self-review remains on backup for the rest of the run.

## Reviewer Action Contract

New Verifier output requires action metadata on every actionable finding:

```yaml
action_id: R2-A1
order: 1
depends_on: []
acceptance_criterion: The command persists the configured backup route
problem: The manifest is updated only in memory
required_fix: Persist active_hands_profile before invoking backup
re_verification:
  - [npx, vitest, run, tests/workflow/runtime-local.test.ts]
```

The runtime validates:

- Action IDs and order numbers are unique within the review revision.
- Orders are positive and contiguous.
- Dependencies reference actions in the same revision.
- Dependencies point only to earlier actions.
- The dependency graph has no cycles.
- Every action contains at least one re-verification command.
- A `request_changes` review contains at least one action.

The saved queue is immutable. A later full Verifier review creates a new queue
revision rather than editing or appending to the previous queue.

For resume compatibility, old saved findings without action metadata are
normalized once by array order and persisted as a normalized queue. Newly
generated Verifier responses must use the strict contract.

## Self-Review Contract

Each self-review pass is a fresh Hands invocation with this structured output:

```yaml
work_item_id: item-1
parent_attempt: 2
pass: 1
active_action_id: R2-A1
findings: []
fixes_applied: []
changed_files: []
commands_attempted: []
remaining_findings: []
ready_for_resolution_check: true
```

The prompt includes:

- The approved work item and acceptance criteria.
- The active Reviewer action, when processing a queue.
- The current diff and latest deterministic verification evidence.
- Completed-action summaries and evidence paths.
- Earlier self-review reports for the same mutation.
- Explicit instruction to inspect independently, fix in-scope findings, and
  report remaining findings honestly.

During initial implementation or a normal free-standing fix, self-review may
fix any defect within the approved work item. During ordered-action processing,
it may fix only:

- Defects directly required to resolve the active action.
- Regressions introduced by the active action’s mutation.
- Changes necessary to preserve previously resolved actions.

Unrelated findings are recorded for the next full Verifier review and do not
expand the active action.

## Per-Mutation Quality Gate

Every successful Hands mutation follows this state machine:

```text
mutation completed
  -> deterministic verification
  -> self-review/fix pass 1
  -> deterministic verification when pass 1 changed files
  -> self-review/fix pass 2
  -> deterministic verification when pass 2 changed files
  -> next independent review gate
```

If configured passes are zero, the runtime proceeds from deterministic
verification to the next independent review gate. A pass is complete only when
its structured report and any required post-pass verification are persisted.

An operational failure does not consume a self-review pass, but blocks the run
because partially applied mutations may exist. Resume uses the persisted
substage and never assumes the pass succeeded.

## Ordered Fix Queue

When a full Verifier review returns actions:

1. Persist the immutable queue revision.
2. Reserve one normal Hands fix cycle when the first action mutation succeeds.
3. Activate the lowest-order action whose dependencies are resolved.
4. Give Hands only the active action, completed-action summaries, approved
   constraints, and relevant evidence.
5. Run the per-mutation quality gate.
6. Run verification for the active action plus all previously resolved actions.
7. Ask a fresh focused Verifier to return `resolved`, `still_open`,
   `blocked`, or `replan_required` for the active action.
8. Unlock the next action only after `resolved`.
9. After the queue completes, run full verification and a new full Verifier
   review.

The complete future queue remains in the ledger but is not placed in the Hands
prompt. This keeps the mutation focused while preserving auditability.

## Focused Resolution Review

The focused Verifier is read-only and receives:

- The immutable active action.
- The before/after diff for its current attempt.
- Deterministic evidence for the active and previously resolved actions.
- Both self-review reports.
- The approved acceptance criterion.

It returns:

```yaml
review_revision: 2
action_id: R2-A1
action_attempt: 1
decision: resolved
evidence_reviewed: []
remaining_problem: null
required_next_fix: null
```

`still_open` must include `remaining_problem` and `required_next_fix`. The same
action repeats until resolved or its configured attempt limit is exhausted.
Focused review cannot add, remove, reorder, or resolve another action.

## Retry Accounting

- A complete action queue is one normal Hands fix cycle.
- The fix cycle is reserved when the first action mutation succeeds.
- Individual action attempts do not consume additional normal fix cycles.
- Each action has its own bounded attempt counter.
- Self-review passes consume neither normal fix attempts nor recovery attempts.
- Operational failures consume no action attempt, but block the run.
- Exhausting an action produces `blocked: action_fix_exhausted`.
- A completed queue followed by another full `request_changes` review advances
  the normal fix-cycle counter and creates a new queue revision.
- After three rejected fix cycles, the existing one-attempt quality recovery
  policy remains the only further recovery tier.

## Usage-Limit Fallback

The existing confirmed-usage-limit classifier and backup catalog validation
apply to:

- Initial Hands implementation.
- Ordered Reviewer action mutations.
- Hands self-review/fix passes.
- Normal fixes and the quality-recovery mutation.

A usage-limited primary invocation does not consume an action attempt or
self-review pass. Backup resumes the same logical substage. If backup is already
active or unavailable, the run blocks without another fallback.

## Resume State

Work-item progress stores:

```yaml
review_revision: 2
queue_state: in_progress
active_action_id: R2-A1
active_action_attempt: 1
completed_action_ids: []
mutation_kind: reviewer_action
self_review_pass: 2
self_review_state: verification_pending
verification_path: verification/issue-1/attempt-2/evidence.json
focused_review_path: null
```

Resume never:

- Regenerates or reorders an existing queue.
- Replays a resolved action.
- Starts a later unresolved action.
- Repeats a completed self-review pass.
- Treats an invocation-start event as a completed pass.
- Resets action attempts or queue-cycle accounting.

Ambiguous partially executed state blocks with an exact artifact and substage
diagnostic.

## Status and Cost Visibility

Local and GitHub status show:

- Queue revision and active action.
- Completed and remaining action counts.
- Active action attempt and limit.
- Current self-review pass and configured total.
- Hands profile/model/reasoning for every invocation.
- Focused Verifier decision.
- Terminal blocker code.

Before queue execution, status reports the worst-case incremental calls:

```text
actions * max action attempts * (1 mutation + self-review passes)
```

For three actions, two action attempts, and two self-reviews, the maximum is 18
Hands calls, six focused Verifier calls, and one final full Verifier call.

## Failure Handling

- Invalid queue structure: `blocked: invalid_reviewer_action_queue`.
- Action limit exhausted: `blocked: action_fix_exhausted`.
- Self-review malformed output: `blocked: invalid_self_review_result`.
- Focused Verifier malformed output: `blocked: invalid_action_resolution`.
- Operational or test infrastructure failure: preserve the existing classified
  blocker and do not advance any queue or pass counter.
- Replan decision: stop queue processing immediately and enter the approved
  replanning path.
- Full Verifier discovers regression after queue completion: create a new queue
  revision; never reopen or mutate the completed revision.

## Testing Strategy

### Configuration

- Accept self-review counts from zero through three.
- Reject counts outside that range.
- Require focused confirmation when ordered actions are enabled.
- Preserve behavior for existing config without `quality_gate`.
- Snapshot policy in run intake.

### Self-review

- Run exactly two fresh invocations after every mutation under the default new
  configuration.
- Use distinct prompts and artifacts for both passes.
- Rerun verification only when a pass changes files.
- Never count self-review as a normal fix or recovery attempt.
- Apply confirmed usage-limit fallback at the same pass and substage.

### Ordered actions

- Reject duplicate IDs, invalid order, forward dependencies, and cycles.
- Expose only the active action to Hands.
- Require active and completed-action regression verification.
- Prevent advancement on `still_open`, operational failure, or malformed
  focused review.
- Advance exactly once on `resolved`.
- Block on action-attempt exhaustion.

### Resume and provenance

- Resume from mutation, verification, self-review pass, post-pass verification,
  and focused review boundaries.
- Never duplicate completed actions or passes.
- Reject mismatched review revision, action ID, attempt, or artifact paths.
- Preserve queue and fix-cycle accounting across interruption.

### End to end

- Initial implementation through two self-reviews to full approval.
- Multiple ordered actions with one `still_open` retry.
- Usage-limit fallback during self-review and action mutation.
- Completed queue followed by a new queue revision.
- Three exhausted fix cycles followed by existing backup quality recovery.
- Exact maximum invocation ceilings and status output.

## Acceptance Criteria

- New configs default to two fresh Hands self-review passes after every mutation.
- Existing configs without the policy retain prior behavior.
- Verifier findings form a validated immutable ordered queue.
- Hands receives and fixes only one active Reviewer action at a time.
- An action advances only after deterministic checks, configured self-reviews,
  regression checks, and focused Verifier `resolved`.
- Queue processing counts as one normal fix cycle.
- Action and self-review limits are bounded, persisted, and resume-safe.
- Usage-limit fallback applies without resetting the logical substage.
- Full verification and full independent Verifier review run after every
  completed queue.
- Local, GitHub, and ledger output expose action, pass, model, evidence, cost,
  and blocker provenance.
